# Validation Documents

> All validation documents for product ideas

## Purpose

This folder contains all validation documents generated during the demand validation phase. Each idea has a complete set of validation documents with consistent naming.

## Document Types

### Required Validation Documents

1. **VALIDATION-PLAN-<idea-slug>.md**
   - Validation plan with 8-12 tests
   - Success/failure thresholds
   - 7-14 day sprint plan
   - Kill/pivot/proceed rules
   - Generated by: `@Demand-Validator`

2. **LANDING-<idea-slug>.md**
   - Landing page copy (hero, offer, transformation)
   - A/B headline variants
   - Visual prompts (Midjourney/Canva)
   - ElevenLabs founder intro script
   - Optional Next.js code
   - Generated by: `@Landing-Builder`

3. **DISTRIBUTION-<idea-slug>.md**
   - Top 15 communities list
   - 7-14 day posting calendar
   - 20 DM outreach scripts
   - Platform-specific content variants
   - Generated by: `@Distribution-Operator`

4. **PRICING-TEST-<idea-slug>.md**
   - 3-5 pricing bands
   - Feature differentiation per tier
   - Fake-door experiments
   - WTP interview questions
   - Generated by: `@Pricing-Tester`

5. **CREATIVE-BATCH-<idea-slug>.md**
   - Glif workflow designs
   - Landing hero + UI mock batch
   - Social card/carousel batch
   - Ad creative batch
   - Generated by: `@Creative-Batch-Operator`

6. **RESULTS-<idea-slug>.md**
   - Validation test results
   - Conversion rates, reply rates, WTP signals
   - Threshold comparisons
   - Daily summaries
   - Generated by: Manual logging + Lindy automation

### Optional Validation Documents

7. **REDTEAM-<idea-slug>.md**
   - Critical analysis of validation strategy
   - Pricing/channel fragility
   - Pivot recommendations
   - Generated by: `@Red-Team-Strategist`

## File Naming Convention

**Format**: `<DOC-TYPE>-<idea-slug>.md`

**Rules**:
- Document type in UPPERCASE (e.g., `VALIDATION-PLAN`, `LANDING`)
- Idea slug in lowercase, hyphenated (e.g., `enterprise-design-system-startups`)
- All documents for the same idea use the same `<idea-slug>`

**Examples**:
- `/docs/validation/VALIDATION-PLAN-enterprise-design-system-startups.md`
- `/docs/validation/LANDING-enterprise-design-system-startups.md`
- `/docs/validation/DISTRIBUTION-enterprise-design-system-startups.md`
- `/docs/validation/PRICING-TEST-enterprise-design-system-startups.md`
- `/docs/validation/CREATIVE-BATCH-enterprise-design-system-startups.md`
- `/docs/validation/RESULTS-enterprise-design-system-startups.md`
- `/docs/validation/REDTEAM-enterprise-design-system-startups.md`

## Idea Slug Requirements

**Idea slugs must be**:
- Lowercase
- Hyphenated (no spaces, underscores, or special characters)
- Descriptive but concise
- Consistent across all documents for the same idea

**Examples**:
- ✅ `enterprise-design-system-startups`
- ✅ `real-estate-investor-flipper-platform`
- ✅ `ai-code-review-documentation-tool`
- ❌ `Enterprise Design System` (uppercase, spaces)
- ❌ `enterprise_design_system` (underscores)
- ❌ `eds` (too short, not descriptive)

## Complete Validation Doc Set

**A complete validation doc set for an idea includes**:
1. VALIDATION-PLAN
2. LANDING
3. DISTRIBUTION
4. PRICING-TEST
5. CREATIVE-BATCH
6. RESULTS (after tests run)
7. REDTEAM (optional but recommended)

**All documents must use the same `<idea-slug>`.**

## Assets Folder

**Visual and audio assets** are stored in:
- `/docs/validation/assets/<idea-slug>/`

**Structure**:
- `/docs/validation/assets/<idea-slug>/landing/hero-images/`
- `/docs/validation/assets/<idea-slug>/social/`
- `/docs/validation/assets/<idea-slug>/ads/`
- `/docs/validation/assets/<idea-slug>/audio/`

## Workflow

See `.cursor/rules/200-playbook-insight-validation.mdc` for the complete validation playbook.

**Validation Phase**:
1. Demand-Validator produces VALIDATION-PLAN
2. Red-Team Strategist produces REDTEAM (optional)
3. Landing-Builder produces LANDING
4. Distribution-Operator produces DISTRIBUTION
5. Pricing-Tester produces PRICING-TEST
6. Creative-Batch-Operator produces CREATIVE-BATCH
7. Execute validation tests
8. Log results in RESULTS
9. Decision: Proceed / Pivot / Kill

**After Validation**:
- If thresholds passed → Proceed to Build Phase
- If thresholds failed → Pivot or Kill
